# unictl v0.7 Execution Plan

## Goal

unictl v0.7 ships an agent-friendly Unity Editor control surface that makes editor lifecycle observable and waitable. Three correctness primitives — heartbeat ABI, PID liveness, and structured `--wait <state>` — replace today's polling-and-guessing patterns. On top of those primitives, the CLI is re-skeletoned to a verb-noun structure with `--describe` introspection, `--json`-by-default output, and a structured error envelope, then opinionated lifecycle-aware feature bundles (`input set`, `deploy android keystore set`, `scripting set`) plus a `settings raw-set --no-warranty` escape hatch are layered on top. Closes/addresses #1, #5, #6.

This revision (revision 2) applies the critic punch list in full. Notable structural changes versus revision 1: spike F.5 moves to Phase 0 because Phase C consumes its result; F.7 gains an explicit dependency edge into Phase D; new spikes F.8 (ABI struct policy) and F.9 (named-pipe vs unix-socket reload parity) are introduced; the heartbeat ABI is redefined as additive-only rather than frozen; runtime.json grows a `terminal_reason` field to remove the teardown race; `/liveness` semantics during reload are pinned down; and `--json` is ON by default on the new verb-noun tree with `UNICTL_HUMAN=1` as the human override.

## Glossary

- **Heartbeat ABI** — the JSON-over-pipe protocol between the Unity-side managed agent and the Rust native bridge that emits the per-frame `{tick, last_heartbeat_ms, state, scene}` record consumed by `/liveness` and `--wait`. ABI is JSON-only; no shared C structs cross the boundary (see F.8).
- **PID liveness** — `runtime.json` keyed by editor PID containing `{pid, started_at_ms, project_root, native_version, schema_version, terminal_reason}`. CLI uses it to distinguish "editor not running" from "editor unreachable".
- **`--wait <state>`** — top-level CLI flag promising that a verb returns only after the editor reaches `idle | playing | compiling | reloading | quit` or a deterministic timeout fires. Internally consumes heartbeat ABI + PID liveness.
- **`--describe`** — verb-level introspection flag emitting a stable JSON document describing arguments, exit codes, output schema, and stability tier. The canonical agent metadata channel for v0.7; `--help --json` becomes an alias and is scheduled for removal in v1.0.
- **Verb-noun tree** — the new CLI shape: `unictl <noun> <verb>` (e.g. `editor compile`, `input set`, `deploy android keystore set`). The legacy v0.6 surface is wrapped in compatibility shims.
- **State envelope** — every `--json` response carries `{ok, code, kind, data, hint}`. `code` is the numeric stable identifier; `kind` is the human-readable slug.
- **Sandbox** — `sandbox/UnictlSmokeProject` is the only Unity project agents may touch for end-to-end work.

## Phase 0 — Pre-C Spikes

Phase 0 holds spikes whose results gate downstream phases. Per critic 2.2 and 2.5, F.5 and F.7 are prerequisites for Phase C and Phase D respectively, so they execute before those phases rather than concurrent with them. F.8 ships before Phase A so the ABI scope is locked.

### F.5 — v0.6 surface compatibility wrapper (executor + api-reviewer, 1d)
Catalog every v0.6 entrypoint, decide for each whether it (a) maps 1:1 to a v0.7 verb-noun pair, (b) maps with a deprecation warning, or (c) becomes a hard error pointing to the new verb. Output: `compat-map.json`. C8 consumes this directly. Pass: every v0.6 surface from the integration tests has a documented disposition.

### F.7 — Push-style heartbeat capability (architect + executor, 1.5d)
Decide whether `--wait` consumes the existing pull `/liveness` route or a new push channel. New native export `unictl_native_subscribe_phase` is the candidate (additive — does not mutate existing exports per critic 1.1). Output: ADR + skeleton export. D3/D4 depend on this output: if push is rejected, D3/D4 fall back to bounded polling against `/liveness`. Pass: ADR signed off and either skeleton export merged or polling fallback documented with budget.

### F.8 — ABI struct layout policy (architect, 0.25d)
Decision: ABI is JSON-over-pipe only. No `repr(C)` structs cross Rust↔C# boundaries in v0.7. Phase A AC inherits this. If a future caller proposes shared structs, that is a v0.8 decision. Output: one-page ADR. Pass: ADR merged before Phase A starts.

### F.9 — Named-pipe vs Unix-socket reload parity (executor + verifier, 0.5d)
`server_windows.rs` (named pipes) and `server_unix.rs` (unix sockets) handle reload differently. Confirm `/liveness` returns the documented reload envelope on both platforms; document any platform-specific quirk. Pass: parity verified on Windows + macOS; if a quirk exists, it is documented in the runbook and surfaced to A4 AC.

### F.1 — Heartbeat ABI perf budget (executor + performance-reviewer, 1d) — DEFERRED TO A2
Two independent ACs (per critic 2.2 split): (a) p99 emit latency < 200µs measured on the sandbox project; (b) allocations < 4KB/s steady-state. Pre-allocated buffer is the fallback if either fails (R3 mitigation). Pass: both ACs green on the sandbox project.

**Deferral**: measurement requires a real heartbeat emitter; an empty stub yields no useful numbers. F.1 is folded into Phase A2 — the A2 PR must include perf-reviewer measurement attached as evidence. F.4 baseline (`F4-transport-baseline.md`) provides regression anchors for D8.

### F.2 — PID liveness file lock semantics (executor, 0.5d)
Decide between flock, lockfile-on-rename, and best-effort writes. Output: locking strategy doc consumed by B1. Pass: chosen strategy passes a 100-iteration concurrent-write test.

### F.3 — `--wait` timeout vs editor reload (test-engineer, 0.5d)
Reload windows can exceed default `--wait` timeouts. Output: recommended default timeout per state plus the override flag. Pass: default + override matrix exists.

### F.4 — Bridge transport throughput (executor + performance-reviewer, 0.5d)
Per critic 2.2, this is measurement work, not defect investigation; `executor + performance-reviewer` replaces the original debugger assignment. Pass: throughput baseline recorded so D8 can detect regression.

### F.6 — Error-registry numeric allocation (architect, 0.5d)
Lock numeric namespaces by domain (editor, deploy, input, scripting, settings, infra). Output: namespace map consumed by C9. Pass: namespaces fit in 16 bits.

**Phase 0 total**: ~6.25d, mostly parallelizable across two executors plus architect.

## Phase A — Heartbeat ABI + `/liveness` route

### A1 — Architecture sign-off (architect, 0.25d)
Per critic 2.3, A1 explicitly assigns the architect a quarter-day to sign off on the JSON-only ABI shape coming out of F.8 and the reload envelope coming out of F.9. AC: ADR merged.

### A2 — Managed emitter (executor, 1.5d)
Implement Unity-side per-frame heartbeat emitter using `Stopwatch.GetTimestamp()` for monotonic timestamps (per R16: wall clock would be unsafe across DST/NTP/VM resume). AC: emitter runs in editor + play mode; matches F.1 perf budget.

### A3 — Rust receiver + `/liveness` route (executor, 1.5d)
Receive heartbeat over JSON-over-pipe. Use `std::time::Instant` (monotonic) on the Rust side. Expose `/liveness` returning `{state, last_heartbeat_ms, since_ms, scene, schema_version}`. AC: route returns within 5ms p99 under steady load.

### A4 — Reload semantics (executor + test-engineer, 1d)
Per critic 1.4: during a domain reload, `/liveness` returns 200 with `{state:"reloading", since_ms:N}`. After 30s without heartbeat, state flips to `unresponsive` (still 200, observable). All other routes return 503 with `{ok:false, code, kind:"editor_reload_active"}` while reload is active. `/liveness` is the only route guaranteed during reload. AC: integration test exercises reload boundary and asserts both paths.

### A5 — Schema versioning AC (writer, 0.25d)
Document the heartbeat schema and the additive-only contract (per critic 1.1): any new export bumps `native_version` minor; consumers must remain forward-compatible with unknown fields. Embedded into A1 ADR.

### A6 — Error path coverage (test-engineer, 0.5d)
AC: every documented error from `/liveness` (timeout, pipe broken, schema mismatch) has an integration test.

### A7 — ABI freeze gate (architect + verifier, 0.25d)
Per critic 1.1, the gate is **additive-only**, not "frozen". Existing exports may not change shape; new exports may be added with a `native_version` minor bump. Consumers must accept unknown fields without erroring. AC: ADR amended; C# and Rust side both compile against a forward-compat fixture.

**Phase A total**: ~5.25d, single executor + architect + test-engineer.

## Phase B — PID liveness

### B1 — `runtime.json` schema (architect, 0.5d)
Schema: `{pid, started_at_ms, project_root, native_version, schema_version, terminal_reason}`. Per critic 1.3, `terminal_reason ∈ {"quit", "crash", "unknown"}`. CLI uses `terminal_reason` to distinguish a clean quit from a crash. AC: schema documented; sample fixture committed.

### B2 — Writer (executor, 1d)
Unity-side writer creates `runtime.json` at editor startup using F.2 locking strategy. AC: file appears within 500ms of editor start.

### B3 — Reader (executor, 0.5d)
CLI-side reader resolves PID + freshness. AC: returns structured result, never panics on partial writes.

### B4a — Clean teardown (executor, 0.5d)
On `EditorApplication.quitting` (and `OnBeforeQuit`), write `terminal_reason: "quit"` **before** any delete attempt. Per critic 1.3, this is the race-free path. AC: integration test confirms a clean quit leaves `terminal_reason: "quit"` observable until the file is removed.

### B4b — Best-effort delete (executor, 0.25d)
After B4a writes, attempt to delete `runtime.json`. Failure is non-fatal; the next startup will overwrite. AC: delete failure path is logged but does not block shutdown.

### B5 — Crash detection (executor, 0.5d)
On startup, if `runtime.json` exists with a stale PID and `terminal_reason != "quit"`, emit `terminal_reason: "crash"` for the previous session into a sidecar before overwriting. AC: kill -9 / taskkill /F is reported as `crash` by the next CLI call.

### B6 — PID guard (executor, 0.5d)
Per critic 2.4, R11 mitigation moves from Phase E to Phase B. Refuse to attach to a `runtime.json` whose `pid` no longer points to a live process matching `project_root`. AC: stale PID is detected and reported as `editor_not_running`.

### B7 — Sandbox tests (test-engineer, 0.5d)
Per critic 3.3, sandbox tests explicitly cover both `taskkill /F /PID` on Windows and `kill -9` on Unix. AC: both platforms produce `terminal_reason: "crash"` on the next CLI invocation.

**Phase B total**: ~4.25d, single executor + test-engineer.

## Phase D — `--wait` integration

D depends on F.7 (per critic 2.2): if F.7 selects push, D3/D4 wire to `unictl_native_subscribe_phase`; otherwise they implement bounded polling against `/liveness`.

### D1 — `--wait <state>` flag plumbing (executor, 1d)
Wire flag into every verb that has a meaningful target state. AC: invalid states fail at parse with `code=64`.

### D2 — State machine (architect + executor, 0.5d)
Document allowed transitions: `compiling → idle`, `idle → playing`, `playing → idle`, `* → reloading → idle`, `* → quit`. AC: state machine documented; transitions tested.

### D3 — Push or poll consumer (executor, 1.5d)
Implements F.7 outcome. AC: either subscribes via the new export and receives transitions within 100ms, or polls `/liveness` at the F.7-documented interval.

### D4 — Wire 6 verbs (executor, 1d)
Per critic 3.1, the verbs are explicitly: `editor compile`, `editor play`, `editor stop`, `editor refresh`, `test editmode`, `test playmode` (count: 6). AC: each verb honors `--wait` with a deterministic default timeout (per F.3 matrix).

### D5 — Timeout handling (executor, 0.5d)
On timeout, return exit code 124 with `kind:"wait_timeout"` envelope. AC: timeout path tested.

### D6 — Reload-aware waiting (executor, 0.5d)
A `--wait` that hits a reload window does not trip the timeout — it transitions to `reloading` and re-arms once `idle` re-appears, capped at the 30s reload ceiling from A4. AC: integration test exercises wait-during-reload.

### D7 — Cancellation (executor, 0.5d)
SIGINT during `--wait` returns code 130 with `kind:"interrupted"`. AC: tested.

### D8 — Crash-mid-wait sandbox test (test-engineer, 0.5d)
Per critic 2.8, when the editor crashes mid-wait (SIGSEGV / Windows access violation, **not** kill -9 — that's covered by B7), CLI returns exit code **125**, not 124. AC: crash injection test green on Windows + Unix; envelope `kind:"editor_crashed"` with `terminal_reason:"crash"` from B5.

**Phase D total**: ~6d.

## Cross-phase concerns

- **Schema versioning policy** (per critic 1.7, applied uniformly to `runtime.json`, error-registry, and `--describe`): adding fields is non-breaking and does **not** bump `schema_version`. Adding required fields, deprecating fields, or changing semantics requires a major bump. Consumers must accept unknown fields. This rule is identical to the heartbeat ABI's additive-only contract from A7; both reference this section.
- **Validation gates**: every PR runs `mise run check`, `mise run check:error-registry`, and `mise run check:meta-guids`. Per critic 3.5, a new `mise run check:heartbeat` aggregator entry joins the suite and runs A6 + B7 + D8 sandbox tests in CI.
- **Commit Discipline**: per `AGENTS.md`, every logical unit ends with a clean tree. Sub-PR boundaries above are also commit boundaries.
- **`.meta` files**: never hand-edited; harness scaffold is the only writer.
- **Sandbox**: all editor-touching tests run against `sandbox/UnictlSmokeProject`. No other Unity project may be opened by automation.
- **`--describe` vs `--help --json`** (per critic 4.0): `--describe` is canonical in v0.7. `--help --json` is documented as an alias and scheduled for removal in v1.0; F1 migration guide states this prominently.
- **Drift gates** (per R17): error-registry drift between Rust CLI and C# emitters is enforced by `check:error-registry`. Both sides must emit `code` + `kind`.
- **Monotonic clocks** (per R16): managed side uses `Stopwatch.GetTimestamp()`; Rust side uses `std::time::Instant`. Wall clock is never used for staleness math.

## Out of scope

- New Unity-version-specific feature bundles beyond `input`, `deploy android keystore`, and `scripting`. Per critic 3.4, this exclusion does not conflict with Phase E: E targets current Unity 6 LTS only and is the documented exception.
- Cross-process orchestration (multiple editors at once). The `feedback_editor_multiproject` rule still applies as a runtime warning, but unictl v0.7 does not coordinate across projects.
- Build-profile builtins are deferred per the existing `project_unictl_roadmap` note.
- IDE integrations beyond CLI invocation.
- Telemetry / metrics export.
- Shared `repr(C)` ABI structs (see F.8); v0.7 is JSON-over-pipe only.

## Constraints

- **Release confirmation**: every release tag, including v0.7.0 itself, requires explicit user confirmation. F6.5 enforces this in `release.ts` so the policy is mechanical, not prose-only (per critic 2.6).
- **`.meta` files**: never hand-edited. Harness scaffold is the only writer.
- **Sandbox isolation**: editor automation runs only against `sandbox/UnictlSmokeProject`.
- **Validation gates**: `mise run check`, `mise run check:error-registry`, `mise run check:meta-guids`, and `mise run check:heartbeat` must be green at every sub-PR boundary.
- **Commit discipline**: clean tree at logical-unit boundaries; sub-PR boundaries are commit boundaries.
- **Additive-only ABI**: post-A7, exports may be added (with native_version minor bump) but never mutated. Consumers must accept unknown exports and unknown fields.
- **Numeric error codes**: 16-bit namespace per F.6; 46 kinds enumerated in `code-allocations.json` per C9.
- **Agent assignments**: architect is READ-ONLY for spec/ADR work; implementation is executor or deep-executor; documentation is writer; reviews use the appropriate `*-reviewer`; release uses git-master + verifier.
